# Mining - AI & Server Connectivity Optimization

현재 서버는 정상 실행되지만, 프론트엔드에서 "서버 연결 오류"가 발생하는 기술적 병목 지점을 분석하고 해결 전략을 수정합니다.

## 🔍 기술적 병목 지점 및 체험 방해 구간

### 1. 하드코딩된 API 주소 (Critical)
- **현상**: `index.html`에서 `fetch('http://localhost:8000/chat')`와 같이 절대 경로를 사용 중.
- **문제**: 사용자가 모바일로 접속하거나, 로컬 IP(`127.0.0.1` vs `localhost`)를 다르게 사용할 때 브라우저 보안 및 DNS 이슈로 연결이 거부될 수 있음.
- **사용자 경험**: "서버가 켜져 있는데 왜 안 되지?"라는 혼란을 야기함.

### 2. 불투명한 에러 핸들링
- **현상**: 네트워크 요청 실패 시 무조건 "서버가 실행 중인지 확인해 주세요"라는 정적 메시지만 노출.
- **문제**: 구글 할당량 초과(429)나 서버측 로직 오류(500)가 발생해도 "서비 미실행"으로 오인하게 만듦.
- **사용자 경험**: 정확한 원인(대기 필요 등)을 알 수 없어 답답함을 느낌.

### 3. Gemini 2.0 Flash 할당량 병목
- **현상**: 무료 티어 사용 시 할당량 초과가 매우 빈번함.
- **문제**: `server.py`에서 429 에러 발생 시 단순 에러를 던지기만 함.
- **사용자 경험**: 대화 중 갑자기 먹통이 됨.

## 🛠 해결 및 고도화 전략 (Stack)

### 1. 통신 레이어 안정화
- **상대 경로 전환**: 모든 `fetch` 주소를 `/chat`, `/hospitals`과 같이 상대 경로로 전환하여 접속 방식에 상관없이 동작하게 함.
- **에러 메시지 구체화**: 백엔드에서 내려주는 에러 메시지(`detail`)를 프론트엔드에서 파싱하여 사용자에게 "잠시 후 다시 시도" 등의 명확한 가이드 제공.

### 2. AI 대화 안정성 확보
- **Retry 로직**: 프론트엔드에서 429 에러 발생 시 짧은 텀을 두고 1회 재시도하는 로직 검토.
- **상태 인디케이터**: AI 분석 중임을 시각적으로 더 명확히 표시하여 사용자 이탈 방지.

### 3. Persistent Summarized Memory (IndexedDB)
- **개별 턴 요약**: 각 대화가 끝날 때마다 `/summarize`를 통해 '상태(Status)' 필드를 추출하여 저장.
- **IndexedDB 활용**: 브라우저 종료 및 새로고침 후에도 `db.js`를 통해 이전 요약 내역을 유지.
- **리포트 최적화**: 전체 대화 로그가 아닌 '요약 데이터'만 병합하여 `/report`로 전송함으로써 효율적인 문맥 전달.
