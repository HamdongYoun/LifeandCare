## 1. 아키텍쳐 툴 및 로직 분석표

| 대응 및 도구(옵션명) | 각 선택지의 장단점 | 독특한 설명(로직 및 장비, 대안 옵션, 대안 옵션에 추가 설계 단계 등) |
|---|---|---|
| **(옵션 A) Flutter + MVVM 아키텍처** | **장점:** 단일 코드베이스로 빠른 생산성, 선언적 UI의 높은 유지보수성. <br> **단점:** 복잡한 네이티브 기능 연동 시 브릿지 개발 부담. | **상태 관리 로직:** View와 ViewModel을 분리하여 챗봇 응답(`OK` vs `EMERGENCY`)에 따른 UI 상태 변화를 예측 가능하게 관리. <br> **대안:** 복잡한 가공이 필요할 경우 백엔드에서 전처리된 데이터만 렌더링하도록 덤(Dumb) UI 전략 사용. |
| **(옵션 B) 위치 기반 실시간 병원 안내 (Geolocator)** | **장점:** 사용자 현재 위치 기반 즉각적 정보 제공. <br> **단점:** 실내 등 GPS 수신 불량 지역에서의 오차 발생. | **권한 로직:** 위치 정보 수집 전 명시적 동의 확인 및 위치 권한 거부 시 '주소 직접 입력'으로 즉시 폴백(Fallback) 처리 설계. |
| **(옵션 C) 음성 인식(STT) 및 외부 앱 연동** | **장점:** 디지털 취약계층 포용성 확보, 응급 상황 대응 속도 단축. <br> **단점:** 외부 스킴(`tel:119`, `map:url`) 호출 시 보안 경고/앱 이탈 우려. | **확장 설계:** 단순히 STT 결과를 보내는 것을 넘어, '오타 교정 및 정문화' 과정을 백엔드와 비동기 통신하여 오인식을 최소화하는 피드백 UI 포함. |

- **메시지 암호화 여부:** 의료 데이터 민감성을 고려한 End-to-End 암호화 적용 여부 결정.

     **음성 키워드 감지:** "살려줘", "도와주세요" 등 특정 데시벨 이상의 긴급 키워드 매칭.

    - 관계 정립: Websocket과 OPUS는 **함께 사용하는 기술**입니다. (택일 대상이 아님)
        -   Websocket: 데이터를 실시간으로 주고받는 **통로(도로)** 역할을 합니다.

        -   OPUS: 그 도로를 지나가는 음성 데이터를 작고 빠르게 압축(차량)하는 규격입니다. 16kHz 모노 샘플링 시 100명 규모에서 서버 대역폭 소모를 최소화합니다.

        - **최적화 시너지:** 0.01 단계에서 Websocket 연결을 맺고, 마이크 입력 데이터를 OPUS로 실시간 인코딩하여 패킷 단위로 전송하면 상용화 채팅 앱과 동일한 성능(600ms~800ms 응답) 가능.

## 1. 프로젝트 목적 및 정의
- 정의: 사용자의 건강 상태 및 증상을 텍스트로 입력받아 AI가 분석하고, 최적의 의료 조치를 추천하는 인공지능 기반 의료 상담 가이드.
- 핵심 가치
    - 증상에 따른 즉각적인 위급도 판단.
    - 사용자의 현재 위치(Lat/Lng) 기반 주변 병원 자동 추천.
    - 응급 상황 시 카운트다운을 통한 119 즉시 연동.

## 2. 해결하고자 하는 핵심 문제 (Pain Points)
- 정보의 불균형: 갑작스러운 아픈 증상 발생 시, 이것이 응급 상황인지 아니면 근처 약국을 가야 할 상황인지 스스로 판단하기 어려움.
- 검색의 피로도: 수많은 포털 검색 결과 중 나에게 맞는 병원을 찾고 위치를 확인하는 과정이 번거롭고 느림.
- 긴급 대응 부재: 119 호출 시 현재 위치를 설명하는 과정에서 발생하는 골든타임 낭비를 방지함.

## 3. 핵심 솔루션 (Key Solutions)
### A. AI 기반 증상 분석 엔진
- 사용자의 입력 텍스트를 분석하여 Text, Hospital Card, Emergency Alert 중 적절한 메시지 타입 결정.
- 증상이 심근경색, 뇌졸중 등 위험 키워드 포함 시 즉시 응급 프로토콜 활성화.

### B. 사용자 위치 기반 병원 카드 (Hospital Card)
- 지도 API와 연동하여 현재 위치에서 가장 가까운 병원의 이름, 주소, 상담 가능 여부를 카드로 렌더링.
- 단순 텍스트가 아닌 인터랙티브 위젯을 통해 직관성 확보.

### C. 119 자동 연동 시스템
- emergencyAlert 타입 메시지 수신 시 메인 화면에 전면 경고 팝업 호출.
- 일정 시간(예: 3초) 카운트다운 후 취소하지 않으면 119로 자동 연결되는 안전 장치 마련.

## 4. 기술 사양 요약 (Tech Specs)
- Frontend: Flutter (MVVM Pattern, Provider, Hive).
- Backend API: Python/Node.js AI Server Interface.
- Persistence: Hive DB (chat_box)를 통한 오프라인 대화 내역 보존.

단순한 채팅을 넘어, 사용자의 생명을 보호하고 더 나은 삶의 질을 제공하는 신뢰도 높은 디지털 의료 가이드로서의 역할 수행.
